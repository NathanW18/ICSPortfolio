What is Object Oriented Programming?

Java is an Object Oriented Programming language. Now, that’s a mouthful to say and a whole lot of typing, so from now on, we’ll refer to it as “OOP.”

There are three features of OOP:

  • Encapsulation
  • Inheritance + Abstraction
  • Polymorphism

In Unit 1, we’ll cover encapsulation. We’ll save the rest for future units.

The opposite of OOP is known as functional programming, where everything is done in one class. No objects are required.

Objects

OOP is meant to recreate real life objects. Objects have methods and attributes. Some methods of a football object could be throwing the football, catching the football, and fixing the football. Some attributes of the football methods could be color, shape, weight, and price. When we change the attributes, we perform actions on the object. Obviously, the object has to exist in the first place, so we instantiate an object.

A class in Java is a blueprint for an object. We use classes to create objects in Java. For example, if we had a Football class, we could do:

Football myBall = new FootBall();

We are making a new Football object called myBall.

Constructor

What if the football we made had a color when we bought it (it wasn’t default)? What if it was shockingly expensive? That’s when we have to use constructors.

Normally, we could just instantiate an object without a constructor, but in case we want to give it custom attributes to start off with, we need a custom constructor to give the object these attributes.

Here is the code for a possible Football object constructor.

public Football (double p) {
    price = p;

    Football myExpensiveBall = new Football(1000.50);

The constructor is a special type of method called when the object is made. It has to be called the same as the class name (i.e., Football). It can take in custom parameters and is overloaded when it does.

Static attributes and methods are not shared by a class’s objects. They are NOT a part of OOP. Thus, a static method can call only other static methods and can only access static variables.

Encapsulation

When we’re coding, we want our data and information to be as safe as possible. That’s why we have encapsulation! If we wanted information in the form of attributes to only be accessible in one class and in no other classes, we would declare that attribute(s) as private. On the other hand, if we didn’t care if other classes could access this information, we would make it public, thus making the attributes editable and viewable. Normally, all instance variables (attributes) are private for security reasons unless there exists a very good reason for it to be public.

Now, here’s a complete class with all information that we discussed, summarized.

The class that I will show here is a class that I commented myself as a project. It has methods and attributes. We use OOP here to show that the class can be made as an object which represents a car in real life.

public class Car {
   private String brand;
   private String model;
   private int kilometres;
   private double gas;
   private double tankCapacity;
   // This method is the constructor, which is called whenever an instance of the class is made.
   // A car is defined by its brand, its model, its gas measure, and its tank capacity when first initialized.
   public Car(String b, String m, double g, double tCap){
      brand = b;
      model = m;
      kilometres = 0;
      gas = g;
      tankCapacity = tCap;
   }
   // This method models the car to drive a distance given as a double.
   // When the car drives several of its attributes change, like the total kilometers as well as the gas level which decreases by the distance divided by 5.
   public void drive(double distance){
      kilometres += distance;
      gas -= distance / 5;
   }
   // This method models a car getting refueled by increasing its gas parameter to the max (the attribute tankCapacity is the max).
   public void gasUp(){
      gas = tankCapacity;
   }
} // Car class

This example shows the various methods, like gasUp as well as how they interact with the attributes, making it a prime example of Object Oriented Programming.

Applications of OOP

  1. Safety because we encapsulate data
    1. Bank accounts, student objects which include name and marks, etc.
  2. Classes describe real life objects
    1. Can “simulate” real-life objects, like a Car object
    2. Can also simulate parts of an object, like a Car Window or an App Login page
    3. Parts and objects can interact with one another. For example, a login screen can interact with an app to change the attribute “isLoggedIn” to true.

Common Mistakes

  1. Accessing private attributes directly. An error will result.
    1. Use an accessor method! Or if you want to modify something, use a mutator method!
  2. Making private attributes public.
    1. It's a security danger! Other classes can change attributes of another class directly!
